OpenWebUI 源码治理：把热改变成可维护系统

今天 OpenWebUI 的关键不是单个 bug，而是治理方式：服务器热修、本地源码、覆盖快照、运行产物、进度表和公开课程必须回到同一条可追踪链路。

今天同步的四类内容

MinerU / RAG

服务器稳定配置沉淀到本地部署模板、说明和私有 env。公开材料只写配置形态，不泄露 key。

视觉与品牌

服务器品牌图标反补到本地静态资源；发现加载页实际吃运行产物后，补覆盖 build 输出并加版本参数。

侧栏微调

把服务器新增的 sidebar polish 文件补进源码静态资源和覆盖快照，并补 Nginx 脱敏示例。

缓存与快捷键

chat 详情从 cache-first 改为 network-first；Mac 刷新快捷键不再被重新生成动作抢走。

课程资产

OpenWebUI fork 课程扩写为长教程，补章节导航和验证门禁，并发布到 Pages。

类型债

项目级检查仍有大量既有类型错误，不能和本次小修混为一谈，需要单独清理线。

治理判断

问题这次判断以后规则
服务器上能跑就算完成吗不算。它只是已验证样本。热改要反补源码、模板、进度、覆盖快照和运行产物。
为什么刷新后问题一会消失一会回来cache-first 把旧 chat JSON 抢先渲染。用户可见详情页默认 network-first，缓存只做网络失败兜底。
为什么品牌图标换了但加载页没换实际服务吃的是 build / SvelteKit 输出里的旧产物。改静态资源时必须核对实际 served URL 的哈希。
还能拆提交吗可以，但不能靠想象拆；要按真实改动面和文件边界拆。先列功能域，再用 Git diff/文件路径拆提交叙事。

阶段地图

阶段 1：服务器热修

线上先跑通，但知识和补丁散在服务器。

阶段 2：源码反补

把热修搬回本地源码和模板。今天处于这一阶。

阶段 3：提交栈整理

按功能拆提交，让 fork 能作为项目经验展示。

阶段 4：全源码部署

用可构建、可回滚、可验证的源码部署替换旧 Docker 热补丁。

阶段 5：上游贡献

挑选通用功能做成最小 PR，不把个人品牌/私有配置带给上游。
