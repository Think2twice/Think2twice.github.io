公众号桥接服务：从回调接通到可用入口

这条线最容易被低估：看起来只是“微信消息转 Telegram”，实际包含公众号被动回复限制、Telegram Web 登录态、菜单/意图路由、凭据安全、网页后台操作可靠性和服务维护。

今天已经跑通的东西

服务骨架

本地项目生成微信 GET 验证、POST 收消息、统一 reply、Telegram webhook、Docker 部署配置和测试。

公网入口

服务部署到服务器本地端口，通过 DNS、Nginx、HTTPS 暴露给公众号后台。公开页不记录真实配置值。

Telegram Web relay

后续改成使用 Telegram Web 登录态，以“用户身份”把查询转给目标 Bot，再把回复带回公众号。

未完成体验

微信被动回复不能等太久，Telegram 命令链接也不能原样变成微信可点击命令，需要后台查询和编号映射。

关键限制

限制今天暴露的现象正确处理
公众号被动回复时间短真实查询超过限制时，用户会感觉“啥也没收到”。秒回“已开始查询”，后台继续处理，结果缓存后让用户发“结果”查看。
Telegram 命令链接不能原样搬到微信Telegram 里的蓝色命令在微信文本里不会自动触发。把命令抽成编号，用户回数字后服务器代发对应命令。
普通聊天不等于资源查询用户发 hello 不应该也触发取书流程。做菜单/按钮/关键词路由，先判断意图再查。
网页后台表单不适合靠系统快捷键在微信后台填表时，快捷键发出系统提示音但没有改字段。用元素级设值、空表单重填或逐字删除，不把 cmd+a 当可靠手段。

推荐下一版形态

  1. 公众号菜单里放“查询资源”。只有点击菜单或使用明确关键词时才进入查询模式。
  2. 用户输入书名后，服务立即回复“已开始查询”。
  3. 后台把书名转给 Telegram Web relay，解析返回列表，把命令映射成编号。
  4. 用户发“结果”拿列表，发“1/2/3”触发对应 Telegram 命令。
  5. 缓存最近一次查询上下文，设置过期时间，避免数字命令串到旧结果。
