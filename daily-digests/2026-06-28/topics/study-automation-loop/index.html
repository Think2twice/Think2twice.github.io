408 学习闭环：单节回填到晚间发布

今天 408 站点的关键变化不是又写了一节内容，而是工作方式切换：白天学习 chat 负责把问题写回小节，晚上 worker 负责把已经落盘的内容同步到正式站。

三层职责

白天：小节学习

每学一节开一个 chat，定位到具体 README，把疑问、类比、图示、错题原因写到最合适的位置。

傍晚：错题沉淀

错题不平铺进主讲义，必要时新建小节专用错题总结，并从主 README 加入口。

晚上：统一发布

worker 只做 sync、构建、链接检查、部署和线上抽查，不重写教材、不 commit、不 push。

今天的样本

样本处理方式沉淀价值
3.7 广域网：为什么 LAN/WAN 不只是范围不同写成“局部共享问题 -> 远距离互联问题 -> 协议体系变化”的学习补洞。以后遇到“主要差异”能从问题空间推答案，不只背选项。
3.7 错题组新建错题总结，主 README 只放入口。个人错题不会挤乱主讲义，但复习能从知识点回到错因。
晚间发布 worker多条执行单元出现构建时序问题，一条成功发布，其余停在部署前。证明“脏改动是输入，不是停止条件”；真正门禁是构建、链接和线上验收。

下一步规则候选

给 408-readme worker 加互斥锁

今天同时有多个自动化执行单元对同一站点目录跑 sync/build，产生了多个不同的瞬时缺文件错误。下一步应改成单入口 orchestrator 或文件锁：同一晚只允许一个发布链路写 `408-study-site`。

把构建 ENOENT 分成两类

如果失败后源文件和同步文件都不存在，是真实缺文件；如果失败后文件存在，可能是构建和同步时序、public 复制或并发清理导致的瞬时缺失。两类不要混在一起修。
