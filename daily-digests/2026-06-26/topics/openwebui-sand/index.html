OpenWebUI / SoruxGPT：功能散点背后的同一条工程线

今天这些小问题看似分散：搜索、模型档位、图标、文件读取、移动端点击、资源包、临时容器、SoruxGPT 护栏。它们共同指向一个问题：本地产品要像 ChatGPT，但不能靠热补丁和偶然配置堆出来。

搜索与来源

截图里出现 `search(...)` 不是正常搜索，而是模型在假装调用工具。处理方式是把搜索变成真实工具链：先执行 RAG web search，拿到真实 URL，再把来源喂给模型回答。YouTube 类请求也要明确走对应搜索源，而不是靠模型猜。

文件任务护栏

SoruxGPT 的文件请求不能因为“生成”两个字误触发图片。护栏的关键不是加几个关键词，而是把文件扩展名、用户意图和可交付物路由到“真实附件”链路，不能生成就明确失败。

模型档位

GPT-5.5 的五档不是五个真实上游模型，而是围绕同一 base model 的 preset 子模型。这样既能给用户清晰选择，也能保留 Arena Model 这类系统模型。

移动端点击

iOS Safari 的首次点击失效不是普通样式问题。桌面和 Android 正常时，要把它当作 WebKit 触控差异处理，并在源码里做只对 iOS 生效的桥接，跳过输入框、编辑器等危险区域。

资源包

默认 Prompts / Skills / Tools 应通过 OpenWebUI API 安装和更新，避免直接改生产数据库。第一版资源包的价值是建立可重复安装的种子，不是一次性把生态塞满。

临时容器边界

本机临时测试容器、目标源码服务和服务器部署必须分清。看到另一个登录页或默认素材时，先查端口、compose 来源和容器名，再决定是否和目标项目有关。

可复用结论

  • OpenWebUI 任务先问“这是源码线、临时容器还是服务器运行面”。
  • 功能修好了不等于完成；还要回迁到本地源码、配置脚本和进度表。
  • 搜索、文件、图片、工具都要先路由，再执行，最后验证真实产物。
