OpenWebUI 散沙合集

今天很多“小问”其实都指向同一个工程事实：OpenWebUI 不是单个模型壳，而是模型、工具、文件、向量库、网关、前端状态和部署数据层共同拼出来的产品。

图片状态卡

前端状态

后端 status event 写入 message.statusHistory；文件事件写入 message.files。最小改动入口在 ResponseMessage/StatusHistory，而不是随意改整个聊天页。

生成文件附件

Files 闭环

PPT/Word/Excel/PDF/zip 不是“模型吐路径”就完成，需要工具生成、文件登记、消息附件和下载权限共同闭环。

模型档位

模型可见性

普通用户看不到多个模型档位，不一定是前端选择器问题，可能是启动补丁或服务端可见模型列表被重写。

RAG/TTS 质量默认

质量优先

embedding 默认应按质量优先收口，低成本模型只能写成备选；脚本应阻止在缺专用 key 时偷用旧 gateway key。

部署数据分层

数据安全

更新功能时替换程序层，不碰持久化数据层。上线前要确认数据卷、数据库、上传文件和知识库目录不会被覆盖。

API 兼容服务

服务器服务

把不同认证/会话形态转成 API 可以服务 OpenWebUI，但公开记录只写状态和卡点，不写真实入口、端口、key 路径。

今天抽出的最小路线

先看事件
状态卡、附件卡和工具输出都要先确认后端事件结构，再决定前端组件入口。
再看持久化
生成文件、RAG、用户数据和模型配置都要确认存在哪里，避免程序更新误伤数据。
最后看提交
每个功能都应该能成为一个逻辑提交：目标、文件、验证、风险都说得清。
