Codex 工作流与提交拆分

当天最重要的 Codex 方法不是“多开几个线程”，而是先确认主线、权限和边界，再把已经做出的复杂功能整理成能长期维护的 commits 与 Skill。

接手前先定位

错误接手的根因不是能力不足，而是没有先确认真正的主线、项目目录、进度表和是否已有线程接管。

自动化可暂停

当用户明确担心自动化乱改时，自动化必须先冻结状态，改为只读核对和收口，不继续扩范围。

Skill 是项目记忆，不是口号墙

OpenWebUI Skill 应记录架构、运行、错误和教学路径；限制性话术或价值判断要先给草案。

大工作区可以拆

混在一起的改动可以按功能重新 stage 成逻辑提交。它不是还原真实时间线，而是整理可读历史。

拆分提交操作图

1. 备份
先建一个保留当前完整状态的备份分支，避免拆提交时丢失工作。
2. 列组
按功能组列计划：文档、网关/RAG、图片、附件、笔记、侧栏、快捷键、品牌资源。
3. 分块 stage
用交互式 stage 或精确路径 stage，每次只放一个功能组。
4. 看 staged diff
提交前确认 staged 区只包含该功能；不相关截图、output、临时文件默认不提交。
5. 再验证
每个逻辑提交至少有能说清的验证：语法、构建、页面、API 或只读证据。
