OpenWebUI 本地源码化与功能闭环

当天最大的工程线不是单个功能，而是把一组“像官方 ChatGPT 一样”的能力拆清：模型、工具、文件、图片、RAG、语音、网关、品牌、部署和 Git 历史。

项目阶段

1. 热修阶段
服务器上先让功能可用：品牌、模型档位、网关、侧栏样式等。风险是改动散落，靠记忆维护。
2. 源码阶段
把热修迁回 fork、本地源码和配置模板；每个功能都写入 PROJECT_PROGRESS 和 Lessons。
3. 闭环阶段
图片状态、生成文件、RAG/TTS、网关后台都要有端到端证据，不只看某个接口 200。
4. 提交阶段
工作区已经足够大，下一步不是继续加功能，而是按逻辑提交拆分，形成可回滚历史。
5. 部署阶段
把程序层更新到服务器，确认不覆盖用户数据、上传文件、知识库和运行配置。

当天关键模块

图片链路

状态事件

已定位状态事件和文件事件如何进入前端消息。风险点是图片请求是否会影响普通聊天上游状态，以及 UI 真实自动出图是否验收。

文件链路

附件下载

Office/PDF/zip 要从工具产物变成 Files 记录，再挂回 message.files。先做最小单类型闭环。

网关后台

管理员入口

第 12 项已回到本地源码：后台页、后端路由和前端 API 同步推进；服务器临时页只作为兼容入口。

RAG/TTS

质量优先

默认策略从“能跑/便宜”改为“质量优先且可验证”。embedding 维度、TTS 可用模型和 key 隔离都写入配置和文档。

品牌面

加载体验

不仅是图标，还包括标题、通知、加载页、运行 DB、build 产物和部署模板。

提交治理

Git 历史

已经不是“能否做功能”的问题，而是如何把功能拆成清楚的 commits，方便未来向上游 fork 解释。
