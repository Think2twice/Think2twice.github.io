大石头专题

OpenWebUI 本地源码化与功能闭环

今天最重的主线：把服务器热补丁、图片链路、文件生成、RAG/TTS、网关管理、模型档位和提交拆分拉进同一条可维护工程链。

今日全景图与阶段判断

1. OpenWebUI 从“像 ChatGPT”走向“解释得清楚的本地产品”

昨天做了什么：图片、文件、RAG、TTS、模型档位、品牌、网关后台和 API 服务都被纳入源码/配置/进度表讨论。

完整阶段地图：临时服务可用 -> 服务器热修 -> fork 源码化 -> 本地可跑 -> 功能闭环 -> 逻辑 commits -> 部署不伤数据 -> 长期维护。

当前阶段判断：处在“源码化 + 功能闭环并行”阶段。能做功能，但还需要把散落改动收成可提交、可部署、可回滚的工程资产。

下一步建议：先用 Git 逻辑提交拆分，把文件附件、图片状态、网关后台、RAG/TTS、品牌资源分批固定下来。

2. Codex 使用方式从“让它继续做”转向“先确认它在哪条线”

昨天做了什么：多次出现“自动化先停”“新接手方向不对”“先读真实主线和进度表”的纠偏。

完整阶段地图：直接下任务 -> 读项目规则 -> 读进度表 -> 判断主线/旁路 -> 决定是否继续改 -> 写入证据。

当前阶段判断：已经开始把“上下文归属”当成工作前置条件。

下一步建议：OpenWebUI 项目 Skill 里加入“接手前五问”：当前主线、进度表、是否有人接管、是否允许改文件、是否允许碰服务器。

3. 文件生成不再被理解成“模型会不会”

昨天做了什么：两次调查都指向同一结论：PPT/Office/zip 的可下载体验是工具链闭环，不是单纯模型回答。

完整阶段地图：模型吐 Markdown 链接 -> 代码执行产物 -> 文件落盘 -> Files 登记 -> message.files -> 前端卡片 -> 下载权限与清理。

当前阶段判断：认知已经从模型能力转向产品链路，但实现仍需一个 MVP。

下一步建议：先只做一种文件类型的最小闭环，别同时追求 PPT/Word/Excel/PDF/zip 全覆盖。

4. 个人自动化能力需要和公开沉淀边界分开

昨天做了什么：有票务 UI 自动化学习样本，也有长期 Skill/规则语言边界的冲突。

完整阶段地图：个人实验 -> 本地学习脚本 -> 安全边界确认 -> 方法层复盘 -> 不公开敏感参数 -> 需要时改成合法学习案例。

当前阶段判断：公开复盘要记录“学到了什么”，但不能把可直接复用的抢购参数、链接和账号写成公开资产。

下一步建议：类似任务只沉淀“元素定位、日志、暂停确认、验收”的自动化工程方法，不沉淀具体抢购目标。

拆分提交

生成文件闭环
