7 月 5 日的状态卡在第三、第四阶段：有的 worker 停在构建链接门禁，有的 worker 通过门禁但停在部署通道。线上旧站可访问，不能证明本轮新内容已发布。

根目标

不是“发布一个网页”，而是让 408 学习材料、刷题入口、错题记录和复盘路径形成稳定闭环，服务长期复习。

工程层

四科源目录写入内容后，统一同步到站点，经过 JSON、构建、链接、legacy assets、部署和线上抽验。

工具层

`sync`、`check:links`、`deploy:phase4`、正式入口 HTTP 抽验、自动化 memory 和 Pages 总账共同构成证据链。

治理层

下一阶段需要 single-flight 发布锁、构建目录稳定策略、SSH/TUN 发布通道预检，以及跨设备做题记录验收。

正确发布流程

触发
白天四科源内容和题库已写入，需要同步到正式站。
动作
预检冲突和 JSON，再 `sync`、`check:links`、`deploy:phase4`。失败即停止并写 memory。
验证
正式入口和本轮变更相关页面/资源返回预期；同步脚本或题库改动还要有功能层验收。
今天保留下来的高价值证据是：同一晚多个 worker 给出了不同失败形态。这不是噪音，而是说明发布系统缺少 single-flight 和构建目录稳定策略。
不要把 `check:links` 失败后的文件存在解释成“可以忽略”。如果失败时读不到、失败后能看到，优先怀疑构建时序、并发清理或文件系统可见性，而不是简单删掉报错。

短期修复

先恢复发布通道，单 worker 重跑 `check:links` 与 `deploy:phase4`，抽验本轮变更页面和资源。

中期规则

给 408 晚间发布增加锁：一个 worker 写，其他 worker 只读等待或归档状态。

长期演进

把做题记录同步、错题、掌握度和复盘图层纳入发布验收，而不是只验静态页面。
