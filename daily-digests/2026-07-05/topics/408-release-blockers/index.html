核心结论：发布失败要先分型，不要直接重试到“看起来好了”。如果失败后文件存在，反而要保留这个证据，因为它指向构建时序或并发写入，而不是简单的源文件缺失。

发布报告必须写四段

预检结果、sync/check 结果、deploy 结果、线上抽验结果。少任意一段都不能叫发布完成。

旧线上仍活着是好事，但不是验收

正式入口返回 200 只能说明旧版本可访问。要验本轮，必须抽查本轮变更对应的页面、资源或脚本。

并发 worker 是风险源

多个 worker 同时写同一站点目录，可能让构建期读到半更新状态。下轮应先加 single-flight。

失败记忆是资产

每个停止点都写入自动化记忆，下一轮才能从“恢复发布通道/消除并发”开始，而不是重读长日志。

公开页已脱敏，只记录发布链路、状态和方法，不写账号、凭据、私密聊天原文或真实服务器细节。
