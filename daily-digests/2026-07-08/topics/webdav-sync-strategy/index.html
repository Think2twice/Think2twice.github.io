WebDAV 同步策略

今天的关键不是“找一个便宜存储”，而是把 AI Notes 的数据面拆成：本地对象模型、同步 adapter、冲突策略、未来托管服务。

结论

首发仍用用户自己的 WebDAV。R2/S3/对象存储、Fay 自己提供托管同步、端到端加密和开发者 SDK 都先作为后续路线，不抢第一版。

触发
用户纠结 iCloud 付费、WebDAV 可用性、对象存储和自建服务器成本。
动作
把“存储介质”与“应用数据同步协议”分开，先做 sync adapter。
验证
ETag/hash、条件写入、tombstone、冲突副本，而不是整包覆盖。

阶段地图

  1. 整包 JSON 推送/拉取：能同步，但容易覆盖。
  2. 对象级同步：每条笔记、附件、版本独立同步。
  3. 冲突 UI：用户看得到冲突，并能选择保留或合并。
  4. 兼容矩阵：覆盖主流 WebDAV 服务和错误恢复。
  5. 托管同步：只有真实用户需求成立后再做，并先设计加密和责任边界。

下一步

优先做冲突列表、保留本地、使用远端、手动合并、恢复删除。不要急着让 Fay 服务器承接用户笔记。
