Sub2API manual deploy

Sub2API手动部署与调试沙盒。

这是一篇脱敏版手动部署教程：先看懂请求怎么从客户端走到容器，再按 Docker Compose、Cloudflare、宝塔 Nginx、HTTPS、Base URL、排错顺序一步步收口。

打开部署沙盒 看手动命令
公网只看域名 端口内网化
Client

客户端填写网关域名、兼容路径和后台生成的 Key。

Cloudflare

DNS 把子域名指向服务器，可开启代理和基础防护。

BaoTa Nginx

负责 80/443、证书、反向代理和流式请求转发。

Local Port

容器只给本机 Nginx 访问，公网不直连原始端口。

公开页脱敏

本页所有 IP、域名、邮箱、路径和密钥都是示例。真实部署时替换为自己的值，但不要把真实服务器信息、Token、密码或后台入口写进公开 GitHub Pages。

先把脑内地图摆正

Sub2API 不是“把某个端口公开出去”这么简单。更稳的理解是：它在服务器里跑一个网关，公网入口只留 HTTPS 域名。

01 Gateway

客户端只认一个统一入口

OpenAI、Claude、Gemini 兼容客户端可以填同一个网关域名，再用 Sub2API 后台生成的 API Key 访问模型。

02 Reverse Proxy

Nginx 把公网流量转回本机

浏览器和 SDK 访问 443，宝塔 Nginx 再转发到 127.0.0.1:8080。

03 Closed Port

原始端口不暴露到公网

如果外部访问 http://203.0.113.10:8080 不通，这是正确方向。真正入口应该是域名和 HTTPS。

用途示例地址说明
后台管理 https://sub2api.example.com/ 登录后台、添加上游账号、创建分组、生成网关 API Key。
OpenAI 兼容 Base URL https://sub2api.example.com/v1 多数 SDK 会在这个基础上拼 /chat/completions、/responses 或 /models。
Claude 兼容 Base URL https://sub2api.example.com 常见路径是 /v1/messages。有些客户端会要求填到 /v1，以客户端说明为准。
健康检查 https://sub2api.example.com/health 部署、反代、证书和 DNS 都通时，应返回类似 {"status":"ok"} 的结果。

Base URL 沙盒

很多调用失败不是服务坏了，而是客户端把路径多拼或少拼了一段。先用沙盒看懂“你填的 Base URL”和“最终请求路径”的关系。

真正手动做一次

            
第一次部署建议先手动跑通，再把稳定命令收进脚本。下面所有命令都用占位符，真实值只放在服务器内部。

          


          

            

              

                
检查 Docker、磁盘和端口

                
先确认服务器能跑容器，根分区有空间，目标端口没有被别的服务占用。

                
docker --version
docker compose version
df -h /
ss -ltnp | sed -n '1,120p'

              

            

            

              

                
准备部署目录

                
把所有持久化数据集中放在一个目录，后面备份、迁移和清理都更容易。

                
mkdir -p /opt/app/sub2api
cd /opt/app/sub2api
curl -fsSL https://raw.githubusercontent.com/Wei-Shaw/sub2api/main/deploy/docker-deploy.sh -o docker-deploy.sh
chmod +x docker-deploy.sh
./docker-deploy.sh

              

            

            

              

                
把容器端口绑定到本机

                
关键点是 127.0.0.1。公网访问不到原始端口，只能从 HTTPS 域名进来。

                
cd /opt/app/sub2api
sed -i 's/^BIND_HOST=.*/BIND_HOST=127.0.0.1/' .env
sed -i 's/^SERVER_PORT=.*/SERVER_PORT=8080/' .env
sed -i 's/^ADMIN_EMAIL=.*/ADMIN_EMAIL=admin@example.com/' .env
sed -i 's/^ADMIN_PASSWORD=.*/ADMIN_PASSWORD=<RANDOM_STRONG_PASSWORD>/' .env
chmod 600 .env

              

            

            

              

                
启动 Sub2API、PostgreSQL 和 Redis

                
先在服务器内部确认服务健康，再去处理域名和证书。

                
cd /opt/app/sub2api
docker compose up -d
docker compose ps
docker compose logs --tail=120 sub2api
curl -fsS http://127.0.0.1:8080/health

              

            

            

              

                
创建 Cloudflare DNS

                
给一个子域名创建 A 记录，指向服务器 IP。公开教程里只写 Token 占位符。

                
export CLOUDFLARE_API_TOKEN="<CLOUDFLARE_API_TOKEN>"
export ZONE_ID="<ZONE_ID>"

curl -X POST "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/dns_records" \
  -H "Authorization: Bearer ${CLOUDFLARE_API_TOKEN}" \
  -H "Content-Type: application/json" \
  --data '{
    "type": "A",
    "name": "sub2api.example.com",
    "content": "203.0.113.10",
    "ttl": 1,
    "proxied": true
  }'

              

            

            

              

                
配置宝塔 Nginx 反代

                
宝塔环境默认改 /www/server/panel/vhost/nginx/*.conf，不要把系统 Nginx 的 /etc/nginx/sites-enabled 当成入口。

                
mkdir -p /www/wwwroot/sub2api.example.com/.well-known/acme-challenge

# 写入 /www/server/panel/vhost/nginx/sub2api.example.com.conf
# 先监听 80，放行 ACME challenge，其余请求反代到 127.0.0.1:8080。

/www/server/nginx/sbin/nginx -t -c /www/server/nginx/conf/nginx.conf
/www/server/nginx/sbin/nginx -c /www/server/nginx/conf/nginx.conf -s reload

              

            

            

              

                
签发证书并切到 HTTPS

                
HTTP challenge 能访问后，再签发证书。证书成功后把 80 跳转到 443。

                
certbot certonly \
  --webroot \
  -w /www/wwwroot/sub2api.example.com \
  -d sub2api.example.com \
  --non-interactive \
  --agree-tos \
  -m admin@example.com

curl -fsS https://sub2api.example.com/health

              

            

          

        


        

          

            
Nginx 模板

            
这个模板只展示结构。实际文件路径、域名、证书路径按你的服务器生成结果替换。

          


          
server {
    listen 80;
    server_name sub2api.example.com;
    underscores_in_headers on;

    location ^~ /.well-known/acme-challenge/ {
        root /www/wwwroot/sub2api.example.com;
        default_type text/plain;
        try_files $uri =404;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl;
    http2 on;
    server_name sub2api.example.com;
    underscores_in_headers on;

    ssl_certificate     /etc/letsencrypt/live/sub2api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/sub2api.example.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;

    client_max_body_size 256M;
    proxy_read_timeout 3600s;
    proxy_send_timeout 3600s;
    proxy_connect_timeout 60s;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_buffering off;
        proxy_cache off;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}


          

            
为什么开 underscores_in_headers on;：一些 API 网关和流式客户端会使用带下划线的请求头。Nginx 默认可能丢弃这类头，导致认证、追踪或客户端识别异常。

          

        


        

          

            
可选：菜单式脚本骨架

            
如果要做成“输入一个命令，然后选 1-7”的终端体验，就把已经手动跑通的阶段封装成函数。

          


          
#!/usr/bin/env bash
set -Eeuo pipefail

APP_DIR="/opt/app/sub2api"
DOMAIN="sub2api.example.com"
ORIGIN_PORT="8080"

menu() {
  printf '%s\n' \
    "1) Check Docker and disk" \
    "2) Prepare deployment directory" \
    "3) Generate or edit .env" \
    "4) Start Docker Compose" \
    "5) Create Cloudflare DNS" \
    "6) Write BaoTa Nginx vhost" \
    "7) Issue HTTPS and verify" \
    "all) Run all steps"
}

step_1_check() {
  docker compose version
  df -h /
}

step_4_start() {
  cd "$APP_DIR"
  docker compose up -d
  docker compose ps
  curl -fsS "http://127.0.0.1:${ORIGIN_PORT}/health"
}

step_7_verify() {
  curl -fsS "https://${DOMAIN}/health"
  ss -ltnp | grep ":${ORIGIN_PORT}"
}


          

            
脚本化的顺序是：先手动跑通，再抽成脚本。不要一开始就把所有步骤塞进脚本，否则失败时很难判断是 Docker、DNS、Nginx 还是证书卡住了。

          

        


        

          

            
调试顺序：从内到外

            
不要一上来猜 Cloudflare。先确认容器，再确认本机端口，然后才看 Nginx、DNS、证书和客户端。

          


          

            

              01 Container
              
容器有没有真的起来

              
cd /opt/app/sub2api
docker compose ps
docker compose logs --tail=120 sub2api
curl -fsS http://127.0.0.1:8080/health

              
通过标准 容器 running，health 返回正常。

            

            

              02 Local Port
              
端口有没有只绑定本机

              
ss -ltnp | grep ':8080'

# 期望看到 127.0.0.1:8080
# 不期望看到 0.0.0.0:8080

              
通过标准 公网扫不到原始端口。

            

            

              03 Nginx
              
宝塔 Nginx 有没有命中

              
/www/server/nginx/sbin/nginx -t -c /www/server/nginx/conf/nginx.conf
/www/server/nginx/sbin/nginx -c /www/server/nginx/conf/nginx.conf -s reload
curl -fsS -I -H 'Host: sub2api.example.com' http://127.0.0.1/

              
通过标准 返回后台、301 或可预期响应。

            

            

              04 Public
              
DNS 和 HTTPS 有没有生效

              
dig +short sub2api.example.com
curl -fsS -I https://sub2api.example.com/
curl -fsS https://sub2api.example.com/health

              
通过标准 HTTPS 可访问，health 正常。

            

          


          

            

              
                
现象优先检查常见原因

              
              
                

                  域名 404
                  宝塔 vhost 目录
                  误写到系统 Nginx、server_name 不匹配、配置未 reload。
                

                

                  证书签发失败
                  /.well-known/acme-challenge/
                  DNS 未生效、Nginx 没放行 challenge、Cloudflare 代理或缓存干扰。
                

                

                  客户端 401
                  Sub2API 后台生成的 API Key
                  用了管理员密码、上游 Key，或者 Key 没绑定正确分组。
                

                

                  客户端路径 404
                  Base URL 多了或少了 /v1
                  OpenAI SDK 多数填到 /v1；部分客户端会自己拼 /v1。
                

                

                  流式输出卡住
                  Nginx buffer、timeout 和升级头
                  未关闭 proxy_buffering，超时太短，或 SSE/WebSocket 头未转发。
                

              
            

          

        


        

          

            
安全收口清单

            
部署成功不是终点。真正上线前，要把公网入口、密钥、日志、备份和合规边界全部收住。

          


          

            

              Must
              
必须做

              
    
                
  • 容器端口绑定到 127.0.0.1。
    • 
                
  • 公网只开放 80/443，通过 HTTPS 访问。
    • 
                
  • .env 权限设置为 600。
    • 
                
  • 管理员密码随机生成，存放在非公开位置。
    • 
                
  • 日志设置轮转，避免容器日志吃满磁盘。
    • 
              

            

            

              Better
              
最好做

              
    
                
  • 后台限制可信 IP，或者加 Cloudflare Access。
    • 
                
  • 开启两步验证，并保存 TOTP 固定密钥。
    • 
                
  • 定期备份应用数据、PostgreSQL 数据和 Redis 数据。
    • 
                
  • 内部脚本可以保存真实变量，公开教程只保留占位符。
    •
