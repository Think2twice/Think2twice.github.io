脱敏声明

本页所有域名、IP、邮箱、路径和密钥都是示例值。真实部署时替换成自己的资产，不要把真实服务器信息写进公开教程。

先看懂它在做什么

Sub2API 是一个 API 网关：客户端只看到你的统一域名和网关 Key，真实上游账号、订阅、调度和统计都在后台管理。

对客户端

OpenAI 兼容客户端通常填：

Base URL: https://sub2api.example.com/v1
API Key:   sk-xxxx-from-sub2api

然后客户端会自动拼出 /v1/chat/completions、/v1/responses、/v1/models 等路径。

对服务器

容器只监听本机端口，公网只走 HTTPS 域名：

Cloudflare
  -> BaoTa Nginx :443
    -> 127.0.0.1:8080
      -> sub2api container

这样外部不能直接访问 http://203.0.113.10:8080。

用途示例地址说明
管理后台 https://sub2api.example.com/ 登录后台、添加上游账号、创建分组和 API Key。
OpenAI 兼容 Base URL https://sub2api.example.com/v1 大多数 OpenAI SDK、ChatBox、Cherry Studio、Open WebUI 等都用这个。
Claude 兼容 Base URL https://sub2api.example.com 请求路径通常是 /v1/messages，是否加 /v1 取决于客户端会不会自动拼路径。
健康检查 https://sub2api.example.com/health 部署、反代、证书和 Cloudflare 都通时应返回 {"status":"ok"}。

菜单式终端脚本的思路

如果想做成“输入一个命令，然后选 1-7”的体验，可以把每个阶段封装成函数。下面是脚本骨架，不包含任何真实密钥。

#!/usr/bin/env bash
set -Eeuo pipefail

APP_DIR="/opt/app/sub2api"
DOMAIN="sub2api.example.com"
ORIGIN_PORT="8080"

step_1_check() {
  docker --version
  docker compose version
  df -h /
}

step_2_prepare() {
  mkdir -p "$APP_DIR"
  cd "$APP_DIR"
  curl -fsSL https://raw.githubusercontent.com/Wei-Shaw/sub2api/main/deploy/docker-deploy.sh -o docker-deploy.sh
  chmod +x docker-deploy.sh
}

step_3_env() {
  cd "$APP_DIR"
  ./docker-deploy.sh
  sed -i 's/^BIND_HOST=.*/BIND_HOST=127.0.0.1/' .env
  sed -i "s/^SERVER_PORT=.*/SERVER_PORT=${ORIGIN_PORT}/" .env
}

step_4_start() {
  cd "$APP_DIR"
  docker compose up -d
  docker compose ps
}

step_5_dns() {
  echo "Use Cloudflare API to create A record: ${DOMAIN} -> 203.0.113.10"
}

step_6_nginx() {
  echo "Write BaoTa Nginx vhost to /www/server/panel/vhost/nginx/${DOMAIN}.conf"
}

step_7_tls_verify() {
  certbot certonly --webroot -w "/www/wwwroot/${DOMAIN}" -d "$DOMAIN"
  curl -fsS "https://${DOMAIN}/health"
}
脚本原则：菜单脚本可以很方便，但第一次部署最好先手动跑一遍。手动跑通后，再把命令整理成脚本，能减少“脚本失败但不知道卡在哪”的痛苦。

手动部署流程

下面用示例域名和示例 IP 展示完整流程。真实操作时，把占位符替换成自己的服务器、域名和 Cloudflare 凭据。

检查服务器资源

先看 Docker、磁盘和端口，避免部署到一半才发现空间不够或端口被占用。

docker --version
docker compose version
df -h /
ss -ltnp | sed -n '1,120p'

准备部署目录

使用本地目录持久化，方便未来整目录备份或迁移。

mkdir -p /opt/app/sub2api
cd /opt/app/sub2api
curl -fsSL https://raw.githubusercontent.com/Wei-Shaw/sub2api/main/deploy/docker-deploy.sh -o docker-deploy.sh
chmod +x docker-deploy.sh
./docker-deploy.sh

调整 .env

把服务端口绑定到本机，设置管理员邮箱、强密码、JWT/TOTP 固定密钥和轻量资源参数。

cd /opt/app/sub2api
sed -i 's/^BIND_HOST=.*/BIND_HOST=127.0.0.1/' .env
sed -i 's/^SERVER_PORT=.*/SERVER_PORT=8080/' .env
sed -i 's/^ADMIN_EMAIL=.*/ADMIN_EMAIL=admin@example.com/' .env
sed -i 's/^ADMIN_PASSWORD=.*/ADMIN_PASSWORD=<RANDOM_STRONG_PASSWORD>/' .env
chmod 600 .env

启动服务

Docker Compose 会启动 Sub2API、PostgreSQL 和 Redis。

cd /opt/app/sub2api
docker compose up -d
docker compose ps
docker compose logs --tail=120 sub2api

创建 Cloudflare DNS

给子域名创建 A 记录，指向服务器。可以开 Cloudflare 代理。

# 示例，不要把真实 token 写进公开脚本
export CLOUDFLARE_API_TOKEN="<CLOUDFLARE_API_TOKEN>"
export ZONE_ID="<ZONE_ID>"

curl -X POST "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/dns_records" \
  -H "Authorization: Bearer ${CLOUDFLARE_API_TOKEN}" \
  -H "Content-Type: application/json" \
  --data '{
    "type": "A",
    "name": "sub2api.example.com",
    "content": "203.0.113.10",
    "ttl": 1,
    "proxied": true
  }'

配置宝塔 Nginx 反代

正式公网入口使用宝塔 Nginx。不要把系统 Nginx 的 /etc/nginx/sites-enabled 当作默认入口。

mkdir -p /www/wwwroot/sub2api.example.com/.well-known/acme-challenge
cat > /www/server/panel/vhost/nginx/sub2api.example.com.conf <<'EOF'
server {
    listen 80;
    server_name sub2api.example.com;
    underscores_in_headers on;

    location ^~ /.well-known/acme-challenge/ {
        root /www/wwwroot/sub2api.example.com;
        default_type text/plain;
        try_files $uri =404;
    }

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_buffering off;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
        client_max_body_size 256M;
    }
}
EOF

/www/server/nginx/sbin/nginx -t -c /www/server/nginx/conf/nginx.conf
/www/server/nginx/sbin/nginx -c /www/server/nginx/conf/nginx.conf -s reload

签发 HTTPS 证书

先确认 HTTP challenge 可访问，再签发证书，最后把 80 跳转到 443。

curl -fsS http://sub2api.example.com/.well-known/acme-challenge/test || true

certbot certonly \
  --webroot \
  -w /www/wwwroot/sub2api.example.com \
  -d sub2api.example.com \
  --non-interactive \
  --agree-tos

certbot certificates -d sub2api.example.com

正式 Nginx 配置模板

证书签发后，把站点配置切成 80 跳 443、443 反代到本机端口。

server {
    listen 80;
    server_name sub2api.example.com;
    underscores_in_headers on;

    location ^~ /.well-known/acme-challenge/ {
        root /www/wwwroot/sub2api.example.com;
        default_type text/plain;
        try_files $uri =404;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl;
    http2 on;
    server_name sub2api.example.com;
    underscores_in_headers on;

    ssl_certificate     /etc/letsencrypt/live/sub2api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/sub2api.example.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    error_page 497 https://$host$request_uri;

    client_max_body_size 256M;
    proxy_read_timeout 3600s;
    proxy_send_timeout 3600s;
    proxy_connect_timeout 60s;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_buffering off;
        proxy_cache off;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
为什么要写 underscores_in_headers on;：一些 API 网关、Codex CLI、流式客户端会使用带下划线的请求头。Nginx 默认可能丢弃这类头，导致认证、追踪或客户端识别异常。

客户端怎么填 Base URL

“域名能打开后台”只是第一层。真正调用模型时，客户端需要使用网关 API Key 和正确的 Base URL。

客户端类型Base URLAPI Key说明
OpenAI SDK / OpenAI 兼容工具 https://sub2api.example.com/v1 sk-xxxx SDK 会拼 /chat/completions 或 /responses。
Claude Code / Anthropic 兼容 https://sub2api.example.com sk-xxxx 常见请求路径是 /v1/messages。有些客户端会要求填到 /v1，按客户端说明调整。
Gemini 兼容 https://sub2api.example.com sk-xxxx 常见路径是 /v1beta/models 和 /v1beta/models/...:generateContent。 
# Python OpenAI SDK 示例
from openai import OpenAI

client = OpenAI(
    api_key="sk-xxxx",
    base_url="https://sub2api.example.com/v1",
)

resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "hello"}],
)
print(resp.choices[0].message.content)

调试顺序：从内到外

排错时不要一上来猜 Cloudflare。按“容器 → 本机端口 → 宝塔 Nginx → DNS → HTTPS → 客户端”的顺序检查。

1. 容器健康

cd /opt/app/sub2api
docker compose ps
docker compose logs --tail=120 sub2api
curl -fsS http://127.0.0.1:8080/health

通过标准 容器 healthy，health 返回 {"status":"ok"}。

2. 端口只监听本机

ss -ltnp | grep ':8080'

# 期望看到：
# 127.0.0.1:8080

通过标准 不应出现 0.0.0.0:8080。

3. 宝塔 Nginx 命中

/www/server/nginx/sbin/nginx -t -c /www/server/nginx/conf/nginx.conf
/www/server/nginx/sbin/nginx -c /www/server/nginx/conf/nginx.conf -s reload
curl -fsS -I -H 'Host: sub2api.example.com' http://127.0.0.1/

通过标准 返回 200、301 或可预期的后台页面响应。

4. DNS 与 Cloudflare

dig +short sub2api.example.com
curl -fsS -I https://sub2api.example.com/
curl -fsS https://sub2api.example.com/health

通过标准 HTTPS 200，health 正常。

现象优先检查常见原因
域名 404 宝塔 vhost 是否写到正确目录 误写到系统 Nginx；server_name 不匹配；配置未 reload。
证书申请失败 /.well-known/acme-challenge/ Cloudflare 代理/缓存、Nginx 没放行 challenge、DNS 未生效。
客户端 401 Sub2API 后台 API Key 用了管理员密码或上游 key，而不是网关生成的 sk-xxxx。
客户端路径 404 Base URL 是否多/少了 /v1 OpenAI SDK 通常填到 /v1；某些客户端会自己拼 /v1。
流式请求卡住 Nginx buffer 和 timeout 未关闭 proxy_buffering，超时时间太短，或 WebSocket/SSE 头未转发。

安全收口清单

部署成功后，真正重要的是把公网面收小，避免“后台能打开”变成“后台裸奔”。

必须做

  • 容器端口绑定到 127.0.0.1。
  • 公网只开放 80/443，通过 HTTPS 访问。
  • .env 权限设置为 600。
  • 管理员密码随机生成并放在非公开位置。
  • 日志设置轮转，避免容器日志吃满磁盘。

最好做

  • 后台只允许少数可信 IP 或加 Cloudflare Access。
  • 开启两步验证，并保存 TOTP 固定密钥。
  • 定期备份 data/、postgres_data/、redis_data/。
  • 把部署命令整理成内部脚本，但公开教程只保留占位符。
合规提醒：API 网关、订阅转 API、账号调度等功能可能受上游服务条款限制。用于学习、内部测试或自用前，应先确认自己对上游账号和调用方式有合规使用权。
