PUBLIC GUIDE

微信公众号到 Telegram 文件中转桥接

用公众号接收书名，用 Telegram Web 用户会话取回资源，再把文件放进 24 小时缓存链接。

用户看到的是一个异步对话，不是长时间等待。

正确路线先定下来

这套桥接服务的核心不是“把文件硬塞回公众号”，而是把公众号做成入口，把 Telegram Web 做成查询执行层，把服务器缓存做成短期文件中转站。

用户侧交互步骤

公众号是被动回复通道，不能让用户卡在一次回调里等 Telegram 查询和文件下载。正确体验是“请求已接收，结果稍后取”。

发送书名

用户输入书名或关键词。服务立刻确认收到，并把查询任务交给后台执行。

取回结果

用户发送“结果”。如果 Telegram 已返回列表，公众号展示编号、书名、作者、年份和格式。

选择编号

用户点击“第 1 本”或直接回复 1。服务把编号映射回 Telegram 返回的资源命令。

请求文件

Telegram Web worker 在目标聊天里发送对应命令，等待文件消息出现，并触发浏览器下载。

返回链接

文件进入服务器缓存目录后，服务生成 https://bridge.example.com/d/<id> 这类 24 小时下载链接。

交互状态机

把每次用户消息都放进状态机，能避免“刚发书名就索要文件”“编号过期还继续下载”“后台还没完成却返回空结果”等错位体验。

等待书名

只接收新的查询关键词

用户还没有有效任务。此时收到普通文本就创建查询任务，收到“结果”则提示先发送书名。

公众号回复 请发送要查找的书名。
后台动作 无后台任务。

为什么用 24 小时缓存中转站

24 小时不是随便选的数字。它把用户当天下载、文件暴露时间、服务器存储压力和清理策略放在同一个可管理边界里。

适配公众号限制

公众号适合返回文本和链接，不适合稳定传递任意格式的大文件。

降低公开暴露面

链接短期有效，过期后文件清理。公开页面也不需要暴露真实资源来源。

避免第三方文件站

不依赖免费网盘、验证码页面或临时上传服务，链路更可控。

方便重新生成

用户错过当天下载窗口，可以重新查询和选择编号，生成新的临时链接。

短期可用，自动清理

缓存目录只保留必要窗口，不做永久文件库。

不要再走的弯路

这些坑看起来像捷径，实际会让链路更脆、更难排查，甚至把私密资产写进公开页面。

弯路 问题 替代做法
用 Bot Token 冒充真人输入框 Bot Token 只能控制那个 Bot 自己，不能代表用户在目标聊天里发消息。 使用已登录的 Telegram Web 用户会话，由 worker 操作真实聊天窗口。
让公众号直接发送任意文件 文件类型、大小和体验都不可控，失败时用户也很难恢复。 返回 24 小时 HTTPS 下载链接，让浏览器处理文件下载。
在微信回调里同步等到底 Telegram 查询和文件下载可能超过回调等待窗口，容易超时。 先确认收到，后台处理，用户再发送“结果”取当前状态。
把免费传文件网站当默认方案 外部站点可能加验证码、限速、改规则，也会扩大隐私风险。 服务器本地缓存，短期链接，过期清理。
按截图猜 Telegram 页面结构 页面样式变化会让截图判断失效，按钮和消息节点可能不在同一层。 用 DOM 和调试接口确认真实节点，再做选择器和回归测试。

上线前验证一条完整链路

只看服务健康是不够的。要从公众号入口跑到下载链接返回，并确认链接真的能下载。

健康检查 
curl -fsS https://bridge.example.com/healthz

✓

✓

✓
下载链接返回 200。过期后要给出明确提示，而不是空白页或服务器错误。
