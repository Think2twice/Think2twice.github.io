长对话大纲与排版：用户问题才是目录

给长聊天增加右侧用户问题大纲，并把助手回答标题字号压到适合阅读的层级。

对应提交 baeadf8a0

这一节先解决什么

长对话真正难找的不是模型回答里的每个小标题，而是用户问过哪些问题、讨论从哪里转向哪里。大纲如果把助手标题也塞进去，会变成噪音目录。

这个功能的核心判断是：聊天页面的导航对象应该是用户意图。用户想回到“我问五子棋小程序那一段”，不是回到助手回答里的某个 Markdown 二级标题。

排版同理。聊天气泡内部不应该用页面 hero 级别的大标题；大字号会抢走整体页面层级，让聊天像文章营销页。

你要先看懂的边界

用户现象

用户看到的是体验问题：按钮点不动、文件下不来、图片跑题、模型不显示、刷新后问题消失。工程上不能只修表象，要追到触发链路。

代码边界

先判断它属于前端组件、后端路由、运行配置、数据持久化还是部署覆盖层。边界判断错了，修复就会漂移。

验收边界

能编译只是第一层。真实验收还要看 API、浏览器行为、刷新后的持久化结果，以及是否误触发昂贵或危险工具。

文档边界

公开教程只写结构、方法和占位符；真实 token、域名、IP、账号、Cookie、服务器路径都不进入 GitHub Pages。

关键文件地图

  • src/lib/components/chat/Messages.svelte：消息区域与大纲联动。
  • src/lib/components/chat/Messages/ResponseMessage.svelte：回答 Markdown 标题样式。

读代码时不要一上来全仓库搜索。先用这张地图定位层，再用 rg 查具体符号，最后才决定改哪个文件。

手工复现流程

  1. 先定义大纲收录对象：只收用户消息，不收助手回答标题。
  2. 给用户问题生成短标题，长标题要截断但不能残字或遮挡。
  3. 点击大纲项滚动到对应用户消息，当前项随滚动状态高亮。
  4. 移动端不强塞右侧栏，避免遮挡聊天。
  5. 回答内部 H1/H2/H3 统一降低字号，保持层级但不压过聊天主界面。

这一节背后的知识课

  • 信息架构里的目录应该服务用户找路，而不是展示所有可解析标题。
  • 聊天页面是工作界面，不是文章页面；标题字号必须受容器限制。
  • 滚动定位要考虑 sticky header、消息高度变化和动态加载。
  • 长文本截断要避免最后一个字被按钮或边框压住。

真正要学会的不是这一组文件名，而是它背后的判断模式。下次换成别的项目时，文件会变，但“先分层、再改动、最后验证真实表面”的顺序不变。

验证门禁

  • 大纲只列用户问题。
  • 点击大纲能滚动到对应问题。
  • 长标题不残字、不溢出。
  • 助手回答标题看起来像聊天正文内层标题，而不是页面主标题。

门禁的意义是把“我觉得可以了”变成“证据显示可以了”。每个提交都应该有自己的最小门禁，而不是全仓库一次性跑到崩。

常见坑和防复发

  • 不要用短聊天误判大纲不存在。
  • 不要把助手所有 Markdown 标题都塞进目录。
  • 不要让移动端大纲遮挡输入框。
  • 不要用 viewport width 动态缩放字体。

给你自己的练习

打开一段 20 轮聊天，手写出 5 个用户问题作为目录。你会发现它比助手标题更适合回看。
