课程首页 00. 版本管理

GitOps-lite

私有 fork 的正确提交栈

当一个 OpenWebUI fork 已经累积大量定制时，最专业的做法不是把所有功能塞进一个提交，也不是维护十几个互相冲突的长期分支，而是整理成一条可阅读、可回滚、可 cherry-pick 的提交栈。

为什么不每个功能一个长期分支

这个项目的功能互相依赖：移动端、图片、文件、笔记、网关、品牌、部署模板都会碰到共享文件。长期分支越多，冲突越多，最后反而不清楚哪一个分支代表真实产品。

正确路线

  1. 保留官方 `upstream/main` 作为干净来源。
  2. 在私有 `origin/codex/fay-openwebui-custom` 上维护 Fay 的产品分支。
  3. 把当前工作树拆成一串逻辑 commit。
  4. 整体 push 到 private fork。
  5. 以后要给官方 PR 时，再从干净 upstream 分支 cherry-pick 单个功能。

手工操作

git branch backup/fay-split-before-YYYYMMDD-HHMMSS
git add -p src/app.html
git diff --cached
git commit -m "feat(mobile): fix iOS Safari tap handling"
git push origin codex/fay-openwebui-custom

本项目采用的规则

本项目采用 GitOps-lite：本地源码是唯一真相，服务器只是运行由源码生成的交付物，项目进度和错误教训写入三件套。
