课程首页 01. 本地开发启动底座

Local Bootstrap

把本机 OpenWebUI 跑成一条稳定链路

这一课对应本地启动底座提交：先让源码、gateway、配置库和模型种子在本机闭环，后面的功能修复才有可靠验证面。

为什么先做启动底座

复杂产品的 bug 往往不是一个文件能解释的。OpenWebUI 的聊天、图片、RAG、语音和模型列表跨越前端、后端、数据库和外部接口。如果本地启动链路不稳定，每次修复都会变成“我到底在测源码、服务器，还是旧缓存”的混乱。

所以这次先建立一个原则：本机默认跑本机 gateway，配置写回本地数据库，模型菜单由脚本种下，服务器只作为之后的部署面和对照面。

四层结构

源码层

`start-local-openwebui.sh` 和 `stop-local-openwebui.sh` 是本地入口，负责启动顺序和进程边界。

配置层

`.env.fay-local` 保存本机环境变量，`sync-local-rag-config.py` 把关键项写进 `webui.db`。

接口层

本机 gateway 监听 `127.0.0.1:13280`，避免默认依赖服务器隧道。

产品层

`apply-local-model-tiers.py` 写入 GPT-5.5 菜单和公开读取权限，让 UI 默认可用。

手工操作顺序

  1. 建 Python 虚拟环境，安装 `backend/requirements.txt`。
  2. 写 `.env.fay-local`，把 OpenWebUI 指向 `http://127.0.0.1:13280/v1`。
  3. 复制 `config.example.json` 到 `.local-run/gateway.config.private.json`，只在本机填真实 key。
  4. 运行 `fay-custom/scripts/start-local-openwebui.sh --screen`。
  5. 检查 `http://127.0.0.1:13280/health` 和 `http://127.0.0.1:8080/health`。

核心判断

`.env` 不是唯一真相。OpenWebUI 的后台配置会落进数据库，所以可靠做法是启动前同步数据库。模型菜单也不是单纯前端文字，它背后有 `model` 表、`access_grant` 权限和用户默认设置。 

bash -n fay-custom/scripts/start-local-openwebui.sh fay-custom/scripts/stop-local-openwebui.sh
python -m py_compile fay-custom/scripts/sync-local-rag-config.py fay-custom/scripts/apply-local-model-tiers.py
