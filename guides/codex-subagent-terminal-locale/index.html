Subagent 怎么用，中文终端乱码怎么修

这页把两个容易混在一起的小问题拆开：什么时候该让 Codex 开子代理，什么时候主线程直接做；以及底部终端把中文路径显示成乱码时，为什么补 UTF-8 locale 就能恢复。

判断是否开 Subagent 修中文路径乱码

先判断，不要一听复杂就开

Subagent 的价值是隔离上下文和并行处理。它不是万能加速按钮，尤其不适合还没想清楚边界的任务。

这次有几个独立问题？
它们会不会改同一批文件？
目标和返回格式清楚吗？
主线程先澄清

一个问题、范围还模糊、或者多个 worker 会改同一块逻辑时，先别急着开子代理。让主 Codex 读规则、定边界、列计划，再决定能不能拆。

提示词：先不要开 subagent。请先把问题拆清楚，列出可独立处理的范围和验证方法。

三种常见叫法

直接把角色、边界、是否改文件、返回格式说清楚，子代理就更像可控流程，不像多开几个聊天窗口。

调查型

多个 explorer 并行看证据

适合读规则、读代码、读日志、读官方文档。只返回证据和结论，不改文件。

实现型

按文件范围分 worker

适合大任务拆成互不重叠的模块。每个 worker 只改自己负责的文件，减少冲突。

审查型

worker 做，reviewer 查

适合先实现，再让 reviewer 分别查规格符合度、代码质量和测试缺口。

请用 subagent。先拆成 2-3 个互不重叠的小任务，每个 subagent 只负责一个范围。最后主线程整合结论并给出验证结果。
请开 3 个 explorer subagent：一个看项目规则，一个看相关代码，一个看测试/报错。只返回证据和结论，不要改文件。

散问题的沉淀路线

这次真正要补上的规则是：有复用价值的散问题，不止写 Markdown 留痕，还要优先变成可浏览 HTML。

1. 当次回答 先把眼前问题解决，不让方法论挡住实操。
2. 判断复用 流程、排障、工具理解、Codex 用法，优先沉淀。
3. 做成 HTML 轻量纸面风格，可加交互沙盒和检查清单。
4. 脱敏验证 不写真实资产；检查布局、链接和索引。
5. 发布归档 更新 GitHub Pages，让 Archive 能回看。

中文路径乱码：先看 locale

如果 macOS Terminal 正常、Codex 底部终端乱码，通常不是文件名坏了，而是嵌入终端启动 shell 时没有拿到 UTF-8。

切换状态

这是安全模拟，不会修改你的电脑。

核心判断 看 `LANG` 和 `LC_CTYPE`，如果是 `C` 或 `POSIX`，中文路径就容易显示成不可读转义。
兜底位置 放在 `~/.zshenv`，因为它比 `~/.zshrc` 更早被 zsh 读取，适合给嵌入式终端兜底。

检查命令

locale | grep -E '^(LANG|LC_CTYPE)='
print -P "%~"

兜底配置

case "${LANG:-}" in
  ""|C|POSIX) export LANG=en_US.UTF-8 ;;
esac

case "${LC_CTYPE:-}" in
  ""|C|POSIX) export LC_CTYPE=en_US.UTF-8 ;;
esac

最后的工作边界

这页本身也示范了新规则：散问题如果已经形成方法，就变成 HTML；但发布前仍要读仓库结构、查 Git 状态、做本地验证。

不要猜发布链路

先查仓库

发布到 GitHub Pages 前，先看 README、目录结构、索引文件和当前 Git 状态。

不要抢首页

弱公开页进 Archive

这类方法页默认 `featured: false`、`homeVisible: false`，方便回看但不挤占首页。

不要暴露隐私

公开页用占位符

服务器、域名、账号、Token、私密聊天原文都不能原样写进公开 HTML。
